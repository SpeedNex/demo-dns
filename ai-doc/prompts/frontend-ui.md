# 十、按钮规范检查

## 1. 按钮尺寸

检查：

* 主按钮高度是否一致
* 次按钮高度是否一致
* 同区域按钮宽度是否统一
* 不同模块按钮尺寸是否保持一致

参考规范：

* Primary Button：高度40px，最小宽度88px
* Secondary Button：高度40px
* Text Button：高度与当前区域保持一致

## 2. 按钮间距

检查：

* 按钮之间间距是否统一
* 按钮与表单距离是否合理
* 按钮与页面边缘距离是否合理

推荐间距：

* 8px
* 12px
* 16px
* 24px

## 3. 按钮优先级

检查：

* 是否只有一个主要操作按钮
* 是否存在多个视觉主按钮
* 主次关系是否明确
* 危险操作是否使用危险样式

推荐层级：

* Primary：主操作
* Secondary：次操作
* Text Button：弱操作
* Danger：删除、禁用等危险操作

# 十七、风险等级定义

请按照以下标准标注风险等级：

## P0 严重问题

满足任意条件即为P0：

* 核心功能不可用
* 数据提交错误
* 数据删除无确认
* 权限可绕过
* 页面严重错位不可操作
* 登录、权限、支付、订单、审核等核心链路异常
* 多语言环境下核心页面无法正常使用

## P1 重要问题

满足任意条件即为P1：

* 功能不完整但有临时替代方案
* 表单校验缺失
* 异常提示不清晰
* 硬编码文案影响多语言上线
* UI规范明显不一致
* 列表、分页、查询存在体验问题
* 部分权限按钮未控制

## P2 优化问题

满足任意条件即为P2：

* 轻微视觉不统一
* 间距不规范
* 文案可优化
* 交互体验可提升
* 代码可维护性一般
* 非核心页面存在轻微问题

# 十八、最终要求

请严格执行以下要求：

1. 不要只给结论，必须给具体问题
2. 不要只说“建议优化”，必须说清楚怎么改
3. 不要只抽样检查，必须尽量全量扫描
4. 不要忽略多语言硬编码问题
5. 不要忽略权限、异常、空状态、Loading、重复提交
6. 不要假设功能存在，必须基于实际页面或代码判断
7. 每个问题必须包含影响和修复建议
8. 能定位到文件和行号的，必须给出文件和行号
9. 不能定位的，必须说明“需进一步验证”
10. 最终必须给出上线建议
11. 如果项目较大，请先输出扫描策略，再分模块输出结果
12. 如果无法一次性完成，请优先输出P0、P1和多语言硬编码问题
