# Feature Start 功能开发提示词

> 用于 AI 执行新功能开发任务（从需求到实现）

## 开发原则

1. **理解需求第一**：先理解业务目标和约束
2. **设计评审**：复杂功能先设计后实现
3. **遵守架构**：遵循现有 ADR 和架构设计
4. **最小化实现**：只实现必需功能，不过度设计
5. **同步文档**：代码和文档同步更新

---

## 开发准备

1. 读取 `project-doc/00-GOAL.md` 了解项目目标
2. 读取 `project-doc/01-ARCHITECTURE.md` 了解架构设计
3. 读取 `project-doc/adr/` 了解架构决策
4. 读取 `project-doc/05-delivery-criteria.md` 了解交付门槛
5. 读取 `rules/coding.md` 了解编码规范
6. 读取 `rules/naming.md` 了解命名规范

## 开发流程

### 阶段 1：需求理解

- 明确功能目标和使用场景
- 明确输入、输出、处理逻辑
- 明确与现有模块的边界
- 识别潜在风险（安全、性能、兼容性）

### 阶段 2：设计评审

**简单功能（< 100 行）**：直接在代码中设计

**复杂功能（≥ 100 行）**：先输出设计方案

```
## 功能设计方案

### 1. 功能概述
[简要描述]

### 2. 核心数据结构
[关键数据结构和接口]

### 3. 模块划分
[新模块与现有模块的关系]

### 4. 实现步骤
1. ...
2. ...

### 5. 风险评估
[可能的风险和应对]
```

### 阶段 3：编码实现

- 遵循编码规范
- 遵循命名规范
- 错误处理完整
- 日志记录适当
- 单元测试（如需要）

### 阶段 4：同步更新文档

**必须更新的文档**：

| 文档 | 更新内容 | 位置 |
|-----|---------|------|
| `specs/{system}/api.md` | 新增/修改的 API 端点、参数、响应 | specs/ |
| `specs/{system}/data-schema.md` / `data-model.md` | 新增/修改的数据表、字段、结构体 | specs/ |
| `03-plans.md` | 当前阶段任务状态 | project-doc/ |
| `04-change-log.md` | 记录本次变更 | project-doc/ |
| `05-delivery-criteria.md` | 如交付门槛或验收方式有变更则同步 | project-doc/ |

**文档更新检查**：
- [ ] specs 文档已更新（端点、参数、响应格式 / 数据模型）
- [ ] 03-plans.md 已标记完成状态
- [ ] 04-change-log.md 已记录本次变更

### 阶段 6：完整性检查与验证

每个功能开发完成后，必须执行以下两步才能结束。

#### 6a. 缺失检查（对照 checklist）

对照 `ai-doc/rules/checklist.md` 逐项检查：

| 检查维度 | 操作 |
|---------|------|
| 数据表完整性 | 涉及的模块是否都有对应的表/字段 |
| API 端点完整性 | 所有需要的端点是否都已定义 |
| 前端页面完整性 | 所有 API 是否有对应的前端页面 |
| 错误处理 | 所有外部接口返回值是否已检查 |
| 权限校验 | 敏感操作是否已校验权限 |
| 文档同步 | 03-plans.md / specs/* / 04-change-log.md 已更新 |

执行动作：
1. **列出本次新增/修改的所有内容**（表、端点、前端页面、服务）
2. **逐项检查是否存在遗漏**（如表缺少索引、端点缺少路由、前端缺少页面）
3. **标记遗漏项**并立即补充

#### 6b. 验证（构建与编译）

逐项执行以下检查：

| 语言/框架 | 检查命令 | 验证项 |
|-----------|---------|--------|
| **Laravel** | `php artisan migrate --force` | 数据库迁移无报错 |
| **Laravel** | `php artisan route:list` | 路由注册正常 |
| **Vue** | `npx vite build` 或 `npm run build` | 前端编译通过 |
| **Go** | `go build ./...` | 所有包编译通过 |
| **Go** | `go vet ./...` | 代码静态分析无问题 |
| **Go** | `go build -o bin/{name} ./cmd/{name}` | 二进制产物生成 |

执行动作：
1. **运行所有适用的验证命令**
2. **修复发现的任何错误**
3. **确认全部通过后记录验证结果**

#### 阶段 6 输出示例

```markdown
| 检查项 | 结果 |
|-------|------|

| API 端点完整性 | ✅ N 个端点全部可用 |
| 前端页面完整性 | ✅ 页面已配齐 |
| Laravel 迁移 | ✅ 通过 |
| Vue 构建 | ✅ 编译通过 |

```

## 实现检查清单

### 必检项

| 检查项 | 说明 |
|-------|------|
| 参数校验 | 用户输入必须校验类型、范围、格式 |
| 错误处理 | 所有可能失败的操作必须有错误处理 |
| 资源释放 | 文件/连接/锁使用后必须释放 |
| 事务一致 | 多表操作必须在事务中 |
| 并发安全 | 共享资源必须有锁保护 |
| 日志记录 | 关键操作必须有日志 |

### 安全检查

| 检查项 | 说明 |
|-------|------|
| 注入防护 | 用户输入不能拼接到 SQL/命令 |
| 权限校验 | 操作前必须校验权限 |
| 敏感数据 | 密钥/Token 不能硬编码 |
| 输入过滤 | 用户输入必须过滤/转义 |

## 注意事项

### 可以做的
- ✅ 先设计后实现
- ✅ 复杂功能拆分
- ✅ 参考现有代码模式
- ✅ 适当预留扩展点

### 不可以做的
- ❌ 不过度设计（不做未来可能用不到的功能）
- ❌ 不改变已有功能的行为
- ❌ 不引入不必要的依赖
- ❌ 不跳过错误处理
